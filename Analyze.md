# StudentHub 学生“一站式”自主管理过程管理系统 · 业务分析报告

---

## 1. 项目定位

### 1.1 业务本质与核心定位
StudentHub 是一套专门面向高校“第二课堂”与学生事务管理的**统一过程管理与综合素质画像平台**。
系统的核心定位是构建 **“一个入口、一套身份、一条主线”** 的一站式学生自主管理数字枢纽：
* **一个入口**：聚合团员发展、社团活动、社区自治、勤工助学四大核心学生事务，避免多系统频繁切换与数据孤岛。
* **一套身份**：以“学生组织身份”为中心，绑定学籍、团籍、社团角色、社区网格岗位及勤工助学岗位等从属关系。
* **一条主线**：以“学生主体 + 过程档案 + 时间戳”为主线，全量沉淀学生在第二课堂与日常事务中的履职与成长数据。

### 1.2 业务痛点与目标（OKR 视角）
系统旨在解决高校学生事务管理中的三大传统痛点：
1. **过程数据散落**：传统的团员发展台账、社团立项、社区巡查、勤工工时分散于各类表格、群聊或纸质记录中，缺乏合规留痕。
2. **规则执行不严**：推优大会到会率、表决通过率、勤工月工时上限、考核合格线等硬性规则缺乏系统级刚性卡控。
3. **画像割裂**：学生在不同组织与岗位中的表现相互孤立，无法形成统一且可信的综合素质评定依据。

**关键业务目标 (OKR)**：
* **过程管理合规化**：团员发展关键节点 100% 留痕；社团活动线上审批闭环率达 99% 以上。
* **自治与安全提效**：社区异常事件平均响应时长缩短 50%；自治队伍考核 100% 线上化。
* **勤工助学精准化**：困难生岗位覆盖率 ≥ 95%；月工时超限（>40h/月）系统级刚性卡控率 100%。
* **画像与决策数据化**：全量过程事件溯源，形成自动化综合素质积分，作为评奖评优唯一客观数据源。

## 2. 用户角色

系统采用 **基于角色的访问控制 (RBAC) + 基于属性的访问控制 (ABAC)** 混合模型。角色纵向分为校级、院系/中间层、学生侧三个层级：

### 2.1 校级管理角色
* **校团委管理员 (`R-SY-LEAGUE` / `R-SY-ADMIN`)**：负责团员发展全局规则制定与终审、全校社团成立/注销/评优终审、星级活动审批及综合素质规则定义。
* **学生处管理员 (`R-SY-STU-AF`)**：负责勤工助学困难认定终审、岗位总量调控、薪酬发放复核、社区自治整体监督与宿舍网格基础字典维护。
* **财务管理员 (`R-SY-FA`)**：负责社团经费报销复核、勤工助学薪酬发放资金对账与流水导出（只读与财务审核）。
* **系统管理员 (`R-SY-OPS`)**：负责基础数据维护、字典配置、角色授权、定时任务监控与系统操作审计日志排查。

### 2.2 院系与中间层角色
* **院系团委书记 (`R-COL-LEAGUE` / `R-COL-ADMIN`)**：负责本院系团员发展审批（积极分子/发展对象/预备团员转正）、院系社团及活动初审。
* **院系辅导员 (`R-COL-COUN`)**：负责本班学生困难认定初审、勤工岗位申请复核、社区晚归/违规事件督办、团员推优大会签字确认。
* **社团指导教师 (`R-COL-TUTOR`)**：负责所指导社团的日常运行监督、活动立项合规性校验、宣传材料审核及年度考核评分。
* **楼管会指导教师 (`R-COL-FLOOR`)**：负责宿舍楼栋自治活动的审核、自治队伍选拔确认、日常巡查复核及重大异常事件响应。

### 2.3 学生侧角色
* **普通学生 (`R-STU-NORM`)**：具备个人事务自助权限，可提交入团申请、加入社团、申请困难认定与勤工岗位、上报社区隐患。
* **团员/团干部 (`R-STU-LEAGUE`)**：参与团支部推优大会表决、撰写季度思想汇报，团支部书记可组织支部会议与录用培养记录。
* **社团干部/社长 (`R-STU-ASSOC`)**：管理社团日常事务，发起社团招新计划、活动立项、现场签到组织及经费报销申请。
* **自治队伍成员 (`R-STU-AUTON`)**：含楼长、楼层长、寝室长，执行日常宿舍卫生巡查、晚归登记、违规电器上报与调解。
* **勤工助学生 (`R-STU-WORK`)**：执行岗位考勤打卡、提交月度自评、查看个人工时明细与薪酬发放进度。

### 2.4 一人多角色与多身份切换机制
系统中单个学生账户可同时兼具团干部、社长、楼层长、勤工助学生等多重身份。系统在用户界面提供**“身份切换器”**，根据用户当前选择的上下文环境动态加载相适应的菜单项与数据作用域。

## 3. 业务流程

### 3.1 全局业务主线
系统以**“统一学生主档案”**为中心，各业务模块独立运行并向主线注入过程事件：
1. **身份绑定**：学生入校后建立基础档案，关联班级、团支部、宿舍床位。
2. **过程履职**：学生参与团员发展、社团活动、社区网格服务或勤工助学，产生带时间戳的审批单、考勤记录、巡查日志。
3. **事件溯源**：各模块在关键节点触发业务事件，发布至总线。
4. **量化计算**：综合素质量化引擎订阅事件，依据评分规则版本实时计算综合分并更新全校排行榜。

---

### 3.2 团员发展业务流程（7 步状态机）
团员发展过程严格执行“申请 → 推优 → 培养 → 发展 → 政审 → 接收 → 转正”单向递进流程：

```
[入团申请] ➔ [支部推优大会] ➔ [培养考察(≥3个月)] ➔ [列为发展对象] ➔ [政审考察] ➔ [发展大会(接收预备团员)] ➔ [转正考察(1年)] ➔ [正式团员]
```

1. **入团申请**：学生提交申请书（政治自述 ≥ 500字，年龄14-28周岁），经辅导员、院系团委、校团委三级审核后进入“推优池”。
2. **推优大会**：团支部召开推优会，系统刚性校验“实到人数 ≥ 应到 2/3”且“赞成票 ≥ 实到 1/2”。通过者确定为入团积极分子。
3. **培养考察**：指定 2 名培养联系人，每月录入培养记录，每季度提交思想汇报（≥1000字，AI查重率卡控）。培养期满 3 个月且团课合格方可晋级。
4. **列为发展对象**：汇总团课证书、群众座谈记录与联系人意见，经支部大会讨论及校团委终审通过。
5. **政审考察**：对发展对象及其主要社会关系出具政审结论（合格/基本合格/不合格），政审合格者方可进入发展程序。
6. **发展大会**：召开接收大会表决通过后，填写《入团志愿书》，校团委批准后成为预备团员，开始 1 年预备期。
7. **转正考察**：预备期满 1 年，提交转正申请与季度考察表，经转正大会表决及校团委终审置为正式团员，发放全国唯一团员证号。

### 3.3 社团活动业务流程

#### (1) 社团全生命周期流程
```
[筹备申请] ➔ [试运行(3个月)] ➔ [注册成立] ➔ [年度评优/星级评定] (异常进入 [评估整顿]) ➔ [注销归档]
```
* **成立**：5-20 名发起人联合起草章程并指定指导教师，经院系团委、校社联、校团委三级审批后颁发编号。
* **换届**：学期末自动提醒换届，新任社长资格审查（无处分）并公示 3 天。
* **评优与整顿**：基于活动质量、签到率、财务规范、会员满意度综合评定 1-5 星级；连续两学期活动不足或评优不过者强制整顿或注销。

#### (2) 社团招新流程
* **计划发布**：社长提交招新计划（目标人数、面试安排），审批通过后发布至招新广场。
* **投递与筛选**：学生在线投递申请（单人同学年限加入最多 3 个社团），社团组织面试并录用。
* **提前结束机制**：具备权限者可手动执行提前结束招新，结束操作不可逆，阻断新投递但保留已投递记录。

#### (3) 活动立项与分级审批流程
* **活动分级卡控**：
  * **A级（跨校/大额预算>1万/涉外）**：指导教师 → 院系 → 校社联 → 校团委 → 校领导审批。
  * **B级（跨院系/500人以上/预算5000-10000元）**：指导教师 → 院系 → 校社联 → 校团委审批。
  * **C级（院系内/100人以上/预算1000-5000元）**：指导教师 → 院系审批。
  * **D级（100人以下/预算<1000元）**：社团指导教师审批。
* **现场开展与签到**：活动现场采用扫码或 GPS 定位签到，迟到 15 分钟计为缺勤。
* **总结与报销**：活动结束后 3 天内提交现场照片与总结，凭立项预算发票发起线上报销审核。

### 3.4 学生社区与自治业务流程

```
[宿舍网格初始化(楼栋-楼层-寝室)] ➔ [自治队伍选举任命] ➔ [日常巡查/晚归/违规上报] ➔ [L1~L4异常事件协同处置] ➔ [学期履职考核]
```

1. **组织结构搭建**：建立“楼栋 — 楼层 — 寝室”三级网格，选举楼长、楼层长、寝室长并公示任命。
2. **日常巡查**：
   * **卫生巡查**：每日打卡评分，连续 3 次不达标自动标记“重点关注寝室”。
   * **晚归与违规**：每日 22:30 后登记晚归；违规电器现场拍照上传并由当事人签字确认。
3. **异常事件协同处置 (L1~L4)**：
   * **L1 (常规报修)**：直接派单至物业，24 小时内处理。
   * **L2 (一般违规/寝室矛盾)**：楼层长与楼管会 4 小时内现场处置。
   * **L3 (聚众饮酒/严重夜不归宿)**：1 小时内联动院系辅导员与楼管会指导教师处理。
   * **L4 (火警/突发事件)**：10 分钟内响应，系统自动触发应急广播、短信及钉钉/企微强提醒，由指导教师与学生处联动结案。
4. **履职考核**：结合巡查完成率、事件结案率、成员满意度进行月度及学期考核，优秀者纳入社区骨干库。

---

### 3.5 勤工助学业务流程

```
[家庭经济困难认定(前置)] ➔ [用人部门岗位发布] ➔ [困难生投递与录取] ➔ [日常打卡考勤与补打卡] ➔ [月度绩效考核] ➔ [系统算薪与发薪]
```

1. **困难认定（前置依赖）**：学生提交家庭经济调查表与证明材料，经班级评议、院系初审、学生处终审后划定困难等级（特别困难/困难/一般困难）。
2. **岗位发布与卡控**：用人部门发布岗位，系统强制卡控“单岗位每周工时 ≤ 20 小时”，困难生优先推荐。
3. **岗位申请与录用**：已认定困难生在线投递（限制在岗岗位数 ≤ 1），用人部门面试录用并由学生确认上岗。
4. **考勤打卡与工时刚性卡控**：
   * 支持一卡通终端打卡或移动端 GPS/人脸识别打卡。
   * 刚性卡控规则：每日 ≤ 8 小时，每周 ≤ 20 小时，**每月 ≤ 40 小时**（超限自动阻断打卡）。漏打卡需辅导员与部门双签补卡。
5. **月度考核与薪酬清算**：次月 5 日前用人部门录入考核分（≥85分全额发薪，60-84分发80%，<60分发50%）。系统依据有效工时、时薪与考核系数自动算薪，经学生处抽查后由财务打入学生个人银行卡。

---

### 3.6 综合素质量化计算流程
1. **事件发布**：四大模块产生业务结果（如转正成功、活动签到完成、异常事件处置优秀、勤工月度考核优秀）。
2. **规则匹配**：量化引擎根据当前生效的规则版本（团内表现30% + 社团25% + 社区20% + 勤工15% + 学业10%）匹配计分逻辑。
3. **分数落库与更新**：生成明细得分记录，实时更新学生综合素质总分，更新全校及院系实时排行榜。

## 4. 功能模块

系统的业务功能划分为 7 大核心模块群：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          StudentHub 功能模块群                          │
├──────────────┬──────────────┬──────────────┬──────────────┬─────────────┤
│ 基础与身份   │ 团员发展     │ 社团活动     │ 社区自治     │ 勤工助学    │
│ (sys/idx)    │ (ty)         │ (st)         │ (sq)         │ (qg)        │
├──────────────┴──────────────┴──────────────┴──────────────┴─────────────┤
│ 综合素质量化分析 (cmp)  │  基础设施与服务 (auth / noti / file / dashboard) │
└─────────────────────────────────────────────────────────────────────────┘
```

### 4.1 基础与身份支撑模块 (`sys` / `idx`)
* **系统配置与字典**：系统参数设置、全局数据字典（政治面貌、困难等级、活动级别、巡查类型）维护。
* **用户与组织树**：包含院系、专业、行政班级、团支部、宿舍楼栋网格的统一树状组织机构管理。
* **学生主档案**：管理学生学籍基础信息、脱敏身份证/手机号、紧急联系人及跨模块履职全景履历。

### 4.2 团员发展模块 (`ty`)
* **入团申请管理**：申请表单提交、资格校验（年龄与政治面貌）、三级审批流转。
* **推优大会管理**：会议发起、应到/实到校验、在线投票计票、推优决议生成。
* **培养考察管理**：培养联系人绑定、月度培养日志、思想汇报在线提交与查重比对、团课学时管理。
* **发展与政审管理**：发展对象资格审查、政审材料上传与结论判定、发展大会与接收表决。
* **转正与花名册**：预备期考察记录、转正申请审批、全国团员证编号分配与团员电子档案花名册。

### 4.3 社团活动模块 (`st`)
* **社团生命周期管理**：社团成立筹备、章程审核、试运行考核、社团换届、年度星级评定与注销。
* **社团招新管理**：招新计划起草与审批、招新广场展示、学生报名投递、面试结果录入、提前结束招新管控。
* **活动立项与审批**：活动表单起草、A/B/C/D 动态分级审批流、应急预案与安全承诺书管理。
* **活动开展与签到**：现场二维码生成、GPS 定位签到校验、迟到/缺勤统计。
* **总结与财务报销**：活动成果与照片总结提交、立项预算匹配校验、发票报销单审核。

### 4.4 学生社区与自治模块 (`sq`)
* **楼栋与宿舍网格**：楼栋-楼层-寝室-床位图形化网格管理、入住与退宿异动管理。
* **自治队伍管理**：楼长/楼层长/寝室长竞选申请、试用期考察、任命与履职档案。
* **日常巡查大厅**：卫生巡查填报、晚归人员登记、违规电器现场拍照取证。
* **异常事件处置中心**：L1-L4 事件分级上报、多部门协同派单处置、应急响应强提醒、结案归档。
* **社区活动与考核**：楼道文化活动审批、寝室学期履职考核与文明寝室评比。

### 4.5 勤工助学模块 (`qg`)
* **困难认定库**：困难认定申请、家庭经济调查表识别、班级评议录入、公示与等级库管理。
* **岗位管理**：用人部门岗位发布、工时与薪酬上限卡控、岗位审查审批。
* **申请与录用**：困难生岗位投递、简历筛选、面试录用与上岗协议确认。
* **考勤与工时管理**：终端/移动端打卡记录、月 40h 工时上限卡控、补卡申请与双签审核。
* **考核与薪酬清算**：月度部门绩效打分、系统自动算薪、发薪明细复核与财务对账导出。

### 4.6 综合素质量化模块 (`cmp`)
* **事件订阅引擎**：监听团员发展、社团、社区、勤工模块的事件总线。
* **规则版本管理**：量化评分公式配置、权重分配（团内/社团/社区/勤工/学业）及版本历代管理。
* **积分与排行榜**：学生综合素质实时得分计算、明细溯源展示、全校/院系/专业多维度排行榜。

### 4.7 辅助与基础设施模块 (`auth` / `noti` / `file` / `dashboard`)
* **鉴权认证 (`auth`)**：登录认证、Token 双令牌机制（Access + Refresh）、黑名单无感刷新。
* **通知中心 (`noti`)**：站内信铃铛通知，集成钉钉、企业微信、短信、邮件多通道推送。
* **文件归档 (`file`)**：异步文件上传、PDF 报表生成、分年月目录归档存储。
* **统一工作台 (`dashboard`)**：针对不同角色聚合待办事项、快捷入口、统计图表与预警告警。

## 5. 数据模型

以下描述系统中核心业务实体之间的逻辑关系与业务属性结构：

### 5.1 基础身份与组织数据模型
* **院系/专业/班级实体**：定义高校三级行政组织结构，维护编码、名称及所属关系。
* **学生主体实体**：核心根实体，维护学号（唯一业务键）、姓名、性别、身份证号（加密）、手机号（加密）、政治面貌、困难等级、当前宿舍床位ID及关联用户账户ID。
* **系统用户与角色实体**：维护用户登录凭证、账户状态，通过用户-角色多对多实体关联 RBAC 角色集合。

### 5.2 团员发展数据模型
* **入团申请实体**：记录申请单号、学生ID、提交日期、政治自述、当前审批状态及各级审批意见。
* **推优大会与投票记录实体**：记录大会编号、应到/实到人数、会议照片路径，并一对多关联候选人的赞成/反对/弃权票数实体。
* **培养考察与思想汇报实体**：记录培养联系人绑定关系、月度考察得分；思想汇报实体记录提交季度、汇报文本、AI查重率及审核状态。
* **政审与发展接收实体**：记录政审结论、函调公文附件；发展大会记录应到实到票数及预备期起止日期。
* **团员花名册实体**：维护正式团员的全国统一团员证号、入团时间、转正时间及组织关系所在团支部。

### 5.3 社团活动数据模型
* **社团主实体**：记录社团编号、名称、所属院系、指导教师ID、社长ID、社团星级（1-5星）及生命周期状态（筹备/试运行/注册/整顿/注销）。
* **社团成员与招新实体**：社团成员实体记录加入时间、职务；招新计划实体记录招新目标人数、面试安排、是否提前结束及结束原因。
* **活动立项与审批实体**：记录活动编号、社团ID、活动名称、级别（A/B/C/D）、预算金额、活动场地、时间及多级审批流记录。
* **活动签到与总结实体**：签到实体记录学生签到时间、地理位置/扫码凭证及迟到状态；总结实体记录活动现场照片、实际参与人数及总结文本。
* **经费报销实体**：记录报销单号、立项ID、报销金额、发票张数及财务审核状态。

### 5.4 社区与自治数据模型
* **楼栋-宿舍-床位网格实体**：树状结构，维护楼栋编号、楼层数、寝室号、床位号及当前入住学生ID。
* **自治岗位任职实体**：记录学生ID、任职职务（楼长/楼层长/寝室长）、管辖楼栋/楼层、任期起止时间及试用期考核结果。
* **巡查记录实体**：记录巡查单号、巡查类型（卫生/晚归/违规电器/隐患）、责任人ID、被查寝室、扣分项及现场照片附件。
* **异常事件实体**：记录事件单号、等级（L1-L4）、发生时间、地点、当事人ID、现场描述、处置状态、协同处置部门及结案归档说明。

### 5.5 勤工助学数据模型
* **困难认定实体**：记录认定编号、学生ID、认定学年、困难等级（特别困难/困难/一般困难）、证明材料附件及公示状态。
* **岗位实体**：记录岗位编号、用人部门、岗位名称、时薪标准、每周工时上限、招聘人数及岗位状态。
* **岗位申请与考勤实体**：申请实体记录学生ID、岗位ID、简历及录用状态；考勤打卡实体记录打卡时间、上下班类型、GPS定位及有效工时。
* **月度考核与薪酬结算实体**：考核实体记录考核月份、部门打分及考核系数；薪酬发放实体记录应发金额、实发金额、银行卡号（脱敏）及财务发放流水。

### 5.6 综合素质量化与日志模型
* **量化评分规则实体**：记录规则版本号、各维度权重占比（团内/社团/社区/勤工/学业）及生效时间。
* **学生量化得分实体**：记录学生ID、规则版本号、团内得分、社团得分、社区得分、勤工得分、学业得分及总分。
* **业务事件日志实体**：记录全局事件ID、触发模块、事件类型、关联学生ID、业务编号、事件载荷JSON及发生时间戳。

## 6. 权限模型

### 6.1 混合权限设计 (RBAC + ABAC)
系统采用 **RBAC（基于角色的访问控制）** 进行功能菜单与基础操作按钮的授权，同时引入 **ABAC（基于属性的访问控制）** 实现细粒度的数据范围隔离与业务状态判定。

```
访问请求 ➔ [RBAC 角色校验] (功能菜单/按钮权限) ➔ [ABAC 属性校验] (院系/组织/数据范围) ➔ 允许/阻断
```

### 6.2 角色分层与操作权限
* **全域管理权限 (`R-SY-ADMIN` / `R-SY-LEAGUE` / `R-SY-STU-AF`)**：拥有全局功能访问权，可进行终审、跨院系数据调阅、规则配置。
* **院系隔离权限 (`R-COL-LEAGUE` / `R-COL-COUN`)**：仅能访问与操作本院系所属学生、班级、社团及社区网格内的数据。
* **特定组织/岗位权限 (`R-COL-TUTOR` / `R-STU-ASSOC` / `R-STU-AUTON`)**：权限严格限定在当前指导的社团、管理的社团或管辖的楼层/寝室范围内。
* **个人自助权限 (`R-STU-NORM` / `R-STU-WORK`)**：仅能查看与提交属于学生本人的申请、档案、考勤与得分记录。

### 6.3 数据边界与作用域隔离
系统在数据读取层自动注入数据作用域过滤条件：
* **本人作用域**：`student_id == current_user.student_id`
* **组织/支部作用域**：`branch_id == current_user.branch_id`
* **社团作用域**：`association_id == current_user.managed_association_ids`
* **楼栋网格作用域**：`building_id == current_user.managed_building_ids`
* **院系作用域**：`college_id == current_user.college_id`

### 6.4 敏感数据脱敏与保护
* **字段级加密存储**：身份证号、手机号、银行卡号在持久层加密存储。
* **展示脱敏**：前端展示时自动脱敏（身份证保留前6后4，手机号保留前3后4，银行卡保留后4位）。
* **敏感权限隔离**：仅具备财务与资助管理角色的用户在特定审核界面允许解密查看完整敏感信息，且每次解密操作强制记录审计日志。

### 6.5 状态机权限硬卡控
所有业务对象的状态推进（如审批通过、驳回、撤销、结案）均绑定特定的角色与前置状态要求。非对应节点授权角色无法调用动作接口，切断绕过前端界面的越权操作。

## 7. 页面职责

### 7.1 工作台与公共页面职责
* **登录页**：提供账号密码认证、身份记住状态、Cookie无感刷新及登录错误提醒。
* **工作台仪表盘 (Dashboard)**：根据当前登录角色动态呈现待办审批事项、今日打卡/巡查提醒、核心业务统计看板及快捷操作入口。
* **通知中心页**：展示站内信列表，提供按已读/未读筛选、分类提醒及点击跳转关联业务单据功能。

### 7.2 学生身份与组织页面职责
* **个人档案页 (My Profile)**：学生展示个人基础信息、政治面貌、履职履历全景图及综合素质得分；管理员可进行信息修正。
* **学生列表与导入页**：管理员按院系/班级检索学生数据，提供批量学生数据导入、映射校验及异常报告下载。
* **组织机构管理页**：维护院系、专业、行政班级及宿舍楼栋的树状结构，分配组织负责人。

### 7.3 团员发展页面职责
* **我的团员发展页（学生端）**：学生视角查看入团发展 7 步履程轨迹、提交入团申请书、上传季度思想汇报、查看培养联系人。
* **入团申请管理页**：辅导员与团委检索本院系申请单，查看政治自述，执行初审与复核操作。
* **推优大会管理页**：团支部书记录入推优大会应到/实到人数、投票结果、上传现场照片并生成推优决议。
* **培养考察与政审页**：培养联系人录入月度表现；管理员查看思想汇报查重结果、上传政审结论公文。
* **团员花名册页**：校团委查看全校正式团员名册、分配团员证号、办理团组织关系转接。

### 7.4 社团活动页面职责
* **社团列表与详情页**：展示全校社团风采、星级排名、展示社团章程及成员名单；社长可编辑社团信息与发起换届。
* **招新广场与申请页**：学生浏览正在招新的社团计划并提交加入申请；社长在招新管理页处理面试录取与提前结束招新。
* **活动立项与审批页**：社长填写活动立项表单（选择级别、预算、应急预案）；审批人员按 A/B/C/D 级别流水线审核。
* **活动签到与总结页**：活动负责人生成现场签到码，查看实时签到人数；活动结束后上传活动总结与发票明细。

### 7.5 社区与自治页面职责
* **楼栋网格图形化页**：以树状网格展现楼栋-楼层-寝室-床位分布及实时入住人员卡片。
* **日常巡查填报页**：楼层长勾选卫生扣分项、登记晚归人员、现场拍照上传违规电器记录。
* **异常事件大厅页**：展示 L1-L4 异常事件列表，提供事件分级过滤、多部门派单协同、现场处置结果录入及结案操作。
* **社区自治考核页**：指导教师对楼长、楼层长、寝室长进行月度及学期履职评分，生成文明寝室榜单。

### 7.6 勤工助学页面职责
* **困难认定申请与管理页**：学生上传困难证明材料；班级与辅导员在线录入评议结果，管理员发布认定公示。
* **岗位广场与申请页**：困难生检索招聘岗位并投递简历；用人部门筛选简历并录入面试结果。
* **考勤打卡与补卡页**：学生执行日常上下班打卡，查看月累计工时；发起漏打卡补卡申请及审核。
* **月度考核与薪酬发放页**：用人部门录入月度绩效分；管理员复核系统自动计算的薪酬明细并导出财务打卡流水。

### 7.7 综合素质与画像页面职责
* **量化得分仪表盘**：学生查看个人在团内、社团、社区、勤工各维度的得分雷达图与得分明细。
* **全校量化排行榜页**：按全校、院系、专业展示综合素质排名，提供评奖评优数据导出。

### 7.8 系统管理与监控页面职责
* **字典与参数管理页**：维护系统基础字典项与业务规则参数。
* **用户与权限管理页**：分配用户角色、重置密码、配置数据作用域。
* **定时任务监控页**：查看系统后台定时任务（如工时汇总、评优计算、日志清理）的运行日志与手动触发执行。

## 8. 页面之间的数据流

系统各页面间的数据流动紧密围绕“学生主身份”与“业务状态机”展开：

```
                             ┌───────────────────┐
                             │  登录认证与工作台  │
                             └─────────┬─────────┘
                                       │ 用户身份与角色路由
                                       ▼
 ┌──────────────────┬──────────────────┬──────────────────┬──────────────────┐
 │   团员发展数据流 │   社团活动数据流 │   社区自治数据流 │   勤工助学数据流 │
 └────────┬─────────┴────────┬─────────┴────────┬─────────┴────────┬─────────┘
          │                  │                  │                  │
          ▼                  ▼                  ▼                  ▼
 [我的团员发展页]     [社团广场/详情页]  [宿舍网格图形页]   [困难认定申请页]
          │ 申请提交         │ 招新投递         │ 巡查/异常上报    │ 审核通过
          ▼                  ▼                  ▼                  ▼
 [入团审批/推优页]    [招新管理/活动立项][异常事件大厅页]   [岗位广场/考勤打卡]
          │ 审核通过         │ 签到/总结提交    │ 处置结案         │ 月度考核/算薪
          ▼                  ▼                  ▼                  ▼
 [培养考察/团员册]    [社团评优/财务报销][社区考核/文明榜]   [薪酬结算/发放页]
          │                  │                  │                  │
          └──────────────────┼──────────────────┴──────────────────┘
                             │ 过程事件触发
                             ▼
                   ┌───────────────────┐
                   │ 综合素质量化得分页 │
                   └─────────┬─────────┘
                             │ 数据汇总
                             ▼
                   ┌───────────────────┐
                   │ 全校排行榜与档案页 │
                   └───────────────────┘
```

### 8.1 认证与权限初始化数据流
1. 用户在**登录页**输入凭证并认证成功，后端返回 Access Token 与 Refresh Token。
2. 前端载入用户基础信息与角色列表，动态构建左侧菜单树与动态路由。
3. 进入**工作台仪表盘**，根据角色身份自动拉取对应的待办列表、预警信息与快捷入口数据。

### 8.2 团员发展业务数据流
1. 学生在**我的团员发展页**提交申请，数据流向辅导员与团委的**入团申请管理页**。
2. 审核通过后，数据进入**推优大会管理页**；团支书录入投票结果后，生成积极分子名单，流转至**培养考察页**。
3. 培养联系人在考察页录入日志，学生提交思想汇报；考察期满后数据流向**发展与政审页**。
4. 政审与发展大会通过后，数据最终流入**团员花名册页**，并向学生**个人档案页**同步团籍状态。

### 8.3 社团活动业务数据流
1. 社长在**社团管理页**起草招新计划，发布后数据同步至**招新广场页**；学生投递后简历流向社长端**招新管理页**，录用后更新**社团详情页**成员名单。
2. 社长在**活动立项页**提交活动表单，数据按 A/B/C/D 级别流入各级审批人的**活动审批页**。
3. 审批通过后，活动进入**活动签到页**；现场签到数据实时生成；活动结束后总结与发票数据流向**经费报销页**。
4. 最终活动履历与报销数据汇总至**社团评优页**，生成社团星级。

### 8.4 社区自治业务数据流
1. 楼层长在**巡查填报页**或**宿舍网格页**录入卫生扣分或异常，数据实时生成异常单流入**异常事件大厅页**。
2. 异常事件在大厅页被分级派发，处置结果同步回**宿舍网格页**（标记重点关注），并流入**社区自治考核页**。
3. 学期考核结果汇总生成文明寝室名单，数据同步更新至学生**个人档案页**。

### 8.5 勤工助学业务数据流
1. 学生在**困难认定页**提交申请，通过辅导员与学生处审核后，学生获得“困难生”标签，解锁**岗位广场页**投递权限。
2. 用人部门在岗位管理页完成录用，数据流向**考勤打卡页**；学生每日打卡生成有效工时记录。
3. 月末工时数据自动汇入**月度考核与薪酬发放页**，用人部门打分后，系统自动算薪并生成财务发薪对账单。

### 8.6 综合素质画像跨页面聚合流
1. 团员发展（转正）、社团（签到/职务）、社区（履职/考核）、勤工（考勤/绩效）各业务页面产生的关键事件，自动发送至后端事件总线。
2. 量化引擎结算后，数据实时流动至**量化得分仪表盘**与**全校排行榜页**。
3. 最终在学生**个人档案页**形成客观全面的综合素质履历画像。

## 9. API 职责

系统 API 遵循 RESTful 规范，以“资源”为核心，辅以 `:action` 端点驱动状态机：

### 9.1 API 整体架构与设计原则
* **统一前缀**：`/api/v1/<module>`。
* **统一封包**：所有响应结构包含状态码、消息文本、业务数据对象及请求追踪 ID。
* **状态机推进**：禁止直接通过数据更新接口修改状态字段，必须调用专用动作端点（如 `/submit`、`/approve`、`/reject`、`/finish`）。

### 9.2 鉴权与系统 API 职责 (`/api/v1/auth`, `/api/v1/sys`)
* **身份认证 API**：处理用户登录、刷新 Token、退出登录及获取当前登录用户全量权限上下文。
* **组织结构 API**：提供院系、专业、班级、团支部、宿舍楼栋树状结构的查询、新增、修改与删除。
* **字典与参数 API**：提供全系统数据字典项的检索与维护。

### 9.3 学生身份与组织 API 职责 (`/api/v1/idx`)
* **学生主档案 API**：提供学生个人履历全景查询、基本信息修改、敏感字段脱敏解密展示。
* **批量导入 API**：处理 Excel 学生数据的异步解析、映射校验、错误报告生成。

### 9.4 团员发展 API 职责 (`/api/v1/ty`)
* **申请流转 API**：处理入团申请提交、撤回、三级审批动作。
* **推优大会 API**：处理推优大会创建、投票数据提交（含刚性票数校验）、推优决议 PDF 生成。
* **培养考察 API**：处理培养联系人绑定、月度培养日志录入、思想汇报提交及查重结果回调。
* **发展与转正 API**：处理政审结论录入、发展大会表决提交、转正申请审批及团员证号分配。

### 9.5 社团活动 API 职责 (`/api/v1/st`)
* **社团管理 API**：处理社团成立申请、试运行评估、换届申请及注销审批。
* **招新管理 API**：处理招新计划起草、投递报名、面试录用及提前结束招新动作（`:finish`）。
* **活动立项 API**：处理活动表单提交、A/B/C/D 分级审批推进、撤回与驳回。
* **签到与总结 API**：处理签到二维码生成、GPS 定位校验、总结报告上传及报销单审核。

### 9.6 社区与自治 API 职责 (`/api/v1/sq`)
* **网格结构 API**：提供楼栋-楼层-寝室-床位网格数据的查询与异动调整。
* **巡查大厅 API**：处理卫生巡查评分、晚归登记、违规电器拍照取证录入。
* **异常事件 API**：处理 L1-L4 异常事件上报、派单协同、现场处置结果补充及指导教师结案。
* **自治考核 API**：处理自治干部学期评分及文明寝室评比数据计算。

### 9.7 勤工助学 API 职责 (`/api/v1/qg`)
* **困难认定 API**：处理家庭经济困难认定申请提交、评议结果录入、认定等级发布。
* **岗位与申请 API**：处理用人部门岗位发布审批、困难生岗位投递与录用确认。
* **考勤打卡 API**：处理打卡数据接收、月 40 小时上限刚性卡控校验、补卡申请与双签审核。
* **考核与发薪 API**：处理月度绩效打分、系统自动算薪计算、发薪明细复核与导出。

### 9.8 综合素质量化 API 职责 (`/api/v1/cmp`)
* **规则配置 API**：提供量化评分规则版本的查询、新增与权重调整。
* **得分与榜单 API**：提供学生个人量化得分明细查询、全校及院系实时排行榜检索。

### 9.9 消息与服务 API 职责 (`/api/v1/noti`, `/api/v1/files`, `/api/v1/dashboard`)
* **通知推送 API**：提供站内信检索、已读标记及第三方通道（钉钉/企微/短信）推送触发。
* **文件归档 API**：处理附件上传、文件类型与大小校验、安全存储路径返回。
* **工作台聚合 API**：按当前登录角色聚合待办事项数量、预警消息及核心统计指标。

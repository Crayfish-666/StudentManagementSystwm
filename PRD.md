## 0. 文档概述与建设目标

### 0.1 建设背景与痛点
高校“第二课堂”与学生日常事务管理长期面临三大核心痛点：
1. **过程数据分散**：团员发展、社团活动、社区自治与勤工助学等业务记录散落在 Excel、微信群与纸质台账中，留痕困难且追溯成本极高。
2. **业务硬卡控缺乏**：推优大会到会率、表决通过率、勤工助学月 40 小时工时上限、活动分级审批等关键合规规则缺乏系统级刚性卡控。
3. **综合素质画像割裂与评语繁重**：学生在不同组织与岗位中的综合表现相互孤立，辅导员在学期末手动撰写大量综合素质测评评语耗时费力。

### 0.2 产品定位与理念
StudentHub 构建**“一个入口、一套身份、一条主线”**的一站式数字枢纽：
* **一个入口**：基于 Vue 3 + Element Plus 的统一前端门户，学生、教师与管理员在单一平台完成全生命周期学生事务。
* **一套身份**：以学籍为核心主线，绑定团籍、社团职务、社区网格岗位及勤工助学岗位等从属组织身份。
* **一条主线**：基于 **“学生主体 + 过程档案 + 时间戳”** 全量沉淀履职事件，通过综合素质量化引擎（CMP）与大模型（LLM）实时生成 AI 综测初稿与可视化画像，为评奖评优提供唯一客观依据。

## 1. 系统架构与技术栈全景（符合答辩考核标准）

* **后端框架**：Java 17 / Spring Boot 3.2.x
* **数据持久层**：MyBatis-Plus 3.5.x + `sqlite-jdbc` 驱动
* **数据库迁移与引擎**：Flyway 自动迁移 + SQLite 3（WAL 模式、强外键约束、5000ms 锁等待）
* **安全与鉴权**：Sa-Token 1.37+（JWT 模式、RBAC + ABAC 细粒度权限、多账号体系、Token 无感刷新）
* **前端框架**：Vue 3.5（Composition API `<script setup>`）+ Vite 5
* **UI 库与可视化**：Element Plus 2.8+ + Pinia 3 + Vue Router 4 + Axios + ECharts 5
* **文件对象存储中间件**：**MinIO 对象存储**（支持分片上传、预签名临时 URL 在线预览、分目录存储、批量打包下载）
* **AI 智能赋能与大模型**：**Spring AI / DeepSeek API 集成**（支持 AI 自动生成学生综合素质测评初稿评语、人工复核覆写、自动分析建议）
* **定时任务调度**：`robfig/cron` / Spring `@Scheduled`（定时核算志愿服务时长、勤工月度薪资与归档）

## 2. 核心用户故事与需求拆解 (User Stories)

### 2.1 基础身份与鉴权模块 (US-AUTH / US-IDX)

#### US-001：多身份统一登录与 Sa-Token 鉴权
* **描述**：作为系统用户（学生/教师/管理员），我希望通过统一登录入口使用学号/工号登录，以便获取对应的系统操作权限。
* **验收标准**：
  1. 接口基于 Sa-Token 颁发 JWT 访问令牌，有效期 30 分钟。
  2. 支持 HTTP-Only Cookie 下发 Refresh Token（有效期 7 天），前端 Axios 拦截器拦截 401 响应并实现透明无感刷新。
  3. 系统自动识别用户角色（如 `R-STU-NORM`, `R-COL-COUN`, `R-SY-LEAGUE`），由 Sa-Token 加载关联的菜单与数据权限隔离。
  4. 支持“身份切换器”，一人多角色用户可在 Vue3 顶栏无缝切换当前上下文环境。

#### US-002：学生全景档案与组织网格维护
* **描述**：作为系统管理员，我希望维护院系-专业-班级-团支部-宿舍网格，并管理学生主档案，以便为各业务模块提供统一数据源。
* **验收标准**：
  1. 支持 Excel 批量导入学生主档案，自动校验学号与身份证号唯一性，生成导入差异报告。
  2. 身份证号、手机号、银行卡号在 SQLite 中采用 AES-256-GCM 加密存储，前端列表自动脱敏展示。
  3. 学生个人档案页展示其团籍状态、社团履历、社区网格岗位、勤工岗位及综合素质量化得分雷达图。

### 2.2 团员发展模块 (US-TY)

#### US-003：入团申请与三级审批
* **描述**：作为普通学生，我希望在系统中提交入团申请书，以便启动团员发展流程。
* **验收标准**：
  1. 系统校验申请人年龄须在 14–28 周岁之间，且当前政治面貌为非团员。
  2. 政治思想自述字数须 ≥ 500 字，同一学生同一时间只能存在一份处于待审批（S1）状态的申请。
  3. 审批流经辅导员初审 → 院系团委复核 → 校团委终审，状态变为 S3 后自动进入“推优池”。

#### US-004：支部推优大会与硬卡控
* **描述**：作为团支部书记，我希望线上召开推优大会并录入表决结果，以便遴选入团积极分子。
* **验收标准**：
  1. 系统刚性校验“实到人数 ≥ 应到人数的 2/3”，否则拒绝提交。
  2. 校验“候选人赞成票 ≥ 实到人数的 1/2”，未达标者无法进入积极分子名单。
  3. 系统自动生成推优决议 PDF 报告并归档至 MinIO，写入状态变更日志。

#### US-005：培养考察与思想汇报查重
* **描述**：作为培养联系人与积极分子，我希望录用培养日志与提交思想汇报，以便满足培养期要求。
* **验收标准**：
  1. 积极分子需绑定 2 名培养联系人，培养期不少于 3 个月，每月需录入至少 1 条考察记录。
  2. 积极分子每季度提交思想汇报（≥1000字），系统调用文本查重逻辑，查重率 >30% 自动打回。
  3. 培养期满且团课结业考试 ≥ 80 分，方可推进至“列为发展对象”节点。

#### US-006：发展大会与预备团员转正
* **描述**：作为校团委管理员，我希望审核发展大会决议与转正申请，以便完成团员身份的正式认定。
* **验收标准**：
  1. 发展大会与转正大会需满足票数刚性卡控。
  2. 预备期须满 1 年方可发起转正申请，转正终审通过后系统自动生成全国唯一团员证号，更新 `idx_student` 中的政治面貌为“正式团员”。

### 2.3 社团活动模块 (US-ST)

#### US-007：社团全生命周期与招新管理
* **描述**：作为社团发起人或社长，我希望申请成立社团并组织招新，以便开展社团活动。
* **验收标准**：
  1. 社团成立需 5-20 名发起人及 1 名在职教职工作为指导教师，经三级审批通过后颁发编号。
  2. 社长起草招新计划，审批通过后发布至招新广场。单一学生同一学年最多加入 3 个社团。
  3. 支持具备权限的角色执行“提前结束招新”动作（`:finish`），操作不可逆并记录原因，阻断新投递但保留已投递记录。

#### US-008：活动立项 A/B/C/D 动态分级审批
* **描述**：作为社长，我希望提交活动立项申请，以便按活动级别完成合规审批。
* **验收标准**：
  1. 系统根据预算金额、参与人数、活动范围自动判定级别：
     * **A级**（跨校/预算>1万/涉外）：指导教师 → 院系 → 校社联 → 校团委 → 校领导。
     * **B级**（跨院系/500人以上/预算5000-10000）：指导教师 → 院系 → 校社联 → 校团委。
     * **C级**（院系内/100人以上/预算1000-5000）：指导教师 → 院系。
     * **D级**（100人以下/预算<1000）：社团指导教师。
  2. A/B 级活动须强制上传应急预案与安全承诺书附件至 MinIO。

#### US-009：现场签到与活动总结报销
* **描述**：作为活动组织者，我希望发起现场签到并提交总结报销，以便闭环活动管理。
* **验收标准**：
  1. 支持二维码生成或 GPS 定位签到，迟到 15 分钟以上自动计为缺勤。
  2. 活动结束后 3 天内须上传现场照片（≥3张）与总结报告。凭立项预算发起线上发票报销审核。

### 2.4 学生社区与自治模块 (US-SQ)

#### US-010：宿舍网格与自治队伍管理
* **描述**：作为社区指导教师，我希望管理“楼栋-楼层-寝室”三级网格并任命自治干部，以便推进宿舍自治。
* **验收标准**：
  1. 页面树状呈现楼栋-楼层-寝室-床位网格及实时入住学生。
  2. 任命楼长、楼层长、寝室长，试用期 1 个月，系统记录履职起止时间。

#### US-011：日常巡查与 L1~L4 异常事件处置
* **描述**：作为楼层长或宿舍管理员，我希望录入日常巡查并在发生突发事件时发起分级响应。
* **验收标准**：
  1. 支持卫生评分、晚归登记、违规电器拍照取证。连续 3 次卫生不达标自动标记“重点关注寝室”。
  2. 异常事件分级响应：L1（常规报修 24h）、L2（一般违规 4h）、L3（严重违规 1h 联动辅导员）、L4（火警/突发事件 10min 启动应急强提醒与短信通知）。
  3. L4 事件必须由指导教师或学生处管理员执行结案。

### 2.5 勤工助学模块 (US-QG)

#### US-012：家庭经济困难认定（前置门禁）
* **描述**：作为困难学生，我希望在线申请困难认定，以便获取勤工助学岗位申请资格。
* **验收标准**：
  1. 学生提交家庭经济调查表与盖章证明，经班级民主评议、院系初审、学生处终审后划分困难等级（特别困难/困难/一般困难）。
  2. 未通过困难认定的学生在投递常规勤工岗位时，系统强制阻断并提示“须先通过困难认定”。

#### US-013：岗位发布与月 40h 工时刚性卡控
* **描述**：作为用人部门与勤工学生，我希望发布岗位与考勤打卡，以便计算薪酬。
* **验收标准**：
  1. 岗位发布卡控：单岗位每周工时 ≤ 20 小时。
  2. 打卡卡控：每日 ≤ 8h，每周 ≤ 20h，**每月 ≤ 40h**。当月打卡累计满 40 小时后，系统硬卡控阻断打卡。
  3. 漏打卡需辅导员与用人部门双签补卡，每月最多 2 次。

#### US-014：月度绩效考核与系统自动算薪
* **描述**：作为用人部门负责人，我希望录入月度考核分，以便系统自动生成薪酬发放表。
* **验收标准**：
  1. 部门在次月 5 日前录入考核分，算薪公式：`月薪酬 = 实际有效工时 × 时薪标准 × 考核系数`（≥85分系数1.0，60-84分0.8，<60分0.5）。
  2. 学生处复核后，生成财务打卡流水，敏感银行卡号脱敏导出。

### 2.6 综合素质量化与 AI 智能模块 (US-CMP / US-AI)

#### US-015：事件溯源与综合素质得分结算
* **描述**：作为学生与管理员，我希望系统自动结算第二课堂综合素质得分，以便查看全景画像与排行榜。
* **验收标准**：
  1. 监听 TY/ST/SQ/QG 模块的业务事件，按当前生效的权重版本（团内30% + 社团25% + 社区20% + 勤工15% + 学业10%）自动计算得分。
  2. 实时更新 `cmp_score` 总分及明细记录，页面展示得分雷达图及全校/院系实时排行榜。

#### US-016：MinIO 分片上传与预签名预览 (考核加分项)
* **描述**：作为用户，我希望上传大文件（如活动视频、证明材料 PDF），以便高效存储与安全预览。
* **验收标准**：
  1. 小文件直传 MinIO，大文件（>10MB）支持前端切片分片上传并由后端自动合并。
  2. 返回带时效的 MinIO 预签名链接（Presigned URL），实现图片与 PDF 安全在线预览。
  3. 支持指定目录打包 Zip 下载与定期清理过期的临时上传文件。

#### US-017：LLM 大模型自动生成综合素质测评初稿与人工复核 (项目 6 加分落地)
* **描述**：作为辅导员，我希望调用 LLM 自动分析学生第二课堂履历并生成综测初稿评语与建议，以便高效完成学期测评。
* **验收标准**：
  1. 系统提取学生当学期在 TY/ST/SQ/QG 的得分与事件明细，拼接 Prompt 发送至 LLM（Spring AI / DeepSeek API）。
  2. LLM 在 5 秒内生成结构化的评语初稿、优点总结与改进建议。
  3. 辅导员可在前端界面对 AI 初稿进行人工复核与覆写保存，最终结果标记“AI初稿+人工覆写”标志。

## 3. 非功能性需求 (NFR)

1. **并发与性能**：在 SQLite WAL 模式与 MinIO 配合下，支持开学季 2000+ 并发在线查询，API 响应时间 < 800ms。
2. **安全性**：Sa-Token 鉴权；敏感字段（身份证/手机号/银行卡）采用 AES-256-GCM 加密落库；MinIO 预签名 URL 访问控制。
3. **数据一致性与审计**：关键状态变更记录到 append-only 的 `event_log` 与 `audit_log`，留痕保存时间 ≥ 5 年。
4. **可靠性**：Flyway 保证数据库版本迁移一致性；`PRAGMA busy_timeout = 5000` 解决写锁冲突。
